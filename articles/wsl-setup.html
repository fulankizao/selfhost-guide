WSL2 lets Windows users run a full Linux kernel without dual-booting. But the default configuration is far from "production-ready" — this guide takes you from installation through step-by-step optimization to using it as a primary development environment.

124| 125|

According to Microsoft's official WSL documentation, over 60% of Windows developers now use WSL for daily Linux development workflows. A 2025 Stack Overflow survey reinforced this, finding that WSL is the most used Linux environment on Windows — surpassing dual-boot and traditional VM setups. This guide helps you go beyond the defaults and build a truly production-grade WSL2 environment.

1. WSL2 vs WSL1

128|

Before we begin, let's clarify the differences between the two versions:

129| 130| 131| 132| 133| 134| 135| 136| 137| 138| 139| 140| 141| 142| 143| 144| 145| 146| 147| 148| 149| 150| 151| 152| 153| 154| 155| 156| 157| 158| 159| 160|
FeatureWSL1WSL2
KernelTranslation layer (not real Linux)Full Linux kernel (VM)
PerformanceFast file ops (cross-OS)Fast syscalls, slow file ops (cross-OS)
CompatibilitySome Linux apps not supportedAlmost all Linux apps supported
NetworkShares Windows IPNAT virtual NIC, needs manual proxy config
Disk FilesDirect access to Windows filesext4.vhdx virtual disk, needs compression
161|

WSL2 is recommended for much better compatibility. If you're still on WSL1, you can upgrade: wsl --set-version <distro_name> 2

2. Basic Configuration

164| 165|

wsl.conf

166|

Create /etc/wsl.conf (this is a WSL-internal config, not a Windows file):

167| 
[boot]
168|systemd=true
169|
170|[user]
171|default=your_username
172|
173|[interop]
174|appendWindowsPath=true
175|

After saving, run wsl --shutdown in PowerShell and re-enter WSL for changes to take effect.

176|

Why do you need systemd? Many Linux services (Docker, sshd, cron) depend on systemd for management. WSL2 doesn't have it by default — adding this line enables it.

177| 178|

Memory and Auto-Reclaim

179|

WSL2 by default can eat a lot of memory (up to 80% of the host). Limit it:

180| 
# Create .wslconfig in your Windows user directory
181|# Path: C:\Users\your_username\.wslconfig
182|
183|[wsl2]
184|memory=4GB
185|processors=4
186|localhostForwarding=true
187|

After saving, run wsl --shutdown and restart WSL for changes to take effect.

188| 189|

Proxy Passthrough

190|

WSL2 uses NAT networking, so proxy software running on Windows is not reachable by default. Add these lines to ~/.bashrc:

191| 
export http_proxy=http://your_proxy_address:7890
192|export https_proxy=http://your_proxy_address:7890
193|export no_proxy=localhost,127.0.0.1,::1,192.168.0.0/16
194|

New terminals will pick it up automatically. If your proxy address changes, you can also write an auto-detection script:

195| 
# Add this to ~/.bashrc to auto-find the Windows proxy
196|host_ip=$(ip route show default | awk '{print $3}')
197|export http_proxy="http://$host_ip:7890"
198|export https_proxy="http://$host_ip:7890"

3. Disk Slim-Down and VHDX Compression

201|

WSL2's ext4.vhdx virtual disk only grows and never shrinks automatically. I've seen a 16GB initial WSL balloon to 80GB+ after six months. According to Microsoft's WSL disk space guide, the VHDX can consume significantly more disk than the actual filesystem usage due to the way ext4 sparse files interact with the virtual disk layer. Regular cleanup + compression is essential.

202| 203|

Step 1: Clean Up Internal Junk

204|

Run inside WSL:

205| 
# Package manager cache — typically frees 200-500MB
206|sudo apt autoremove -y && sudo apt clean
207|
208|# Docker leftovers (if any)
209|docker system prune -af
210|
211|# pip/npm cache — pip cache can be 100-300MB
212|rm -rf ~/.cache/pip/
213|npm cache clean --force
214|
215|# systemd logs — logs can accumulate several GB
216|sudo journalctl --vacuum-time=7d
217|sudo journalctl --vacuum-size=200M
218| 219|

Step 2: Compress VHDX

220|

Run in Windows PowerShell (as Administrator):

221| 
# 1. Shut down WSL first
222|wsl --shutdown
223|
224|# 2. Find the ext4.vhdx path
225|# Usually at: C:\Users\your_username\AppData\Local\Packages\distro_name\LocalState\
226|# You can find it with:
227|dir C:\Users\your_username\AppData\Local\Packages\*\LocalState\ext4.vhdx
228|
229|# 3. Compress with diskpart (admin PowerShell)
230|diskpart
231|    select vdisk file="C:\Users\your_username\AppData\Local\Packages\CanonicalGroupLimited.Ubuntu24.04LTS_79rhkp1fndgsc\LocalState\ext4.vhdx"
232|    attach vdisk readonly
233|    compact vdisk
234|    detach vdisk
235|    exit
236|

In practice this can compress an 80GB VHDX down to 20GB. Recommended to do this monthly.

4. File Interop Tips

239|

File access between WSL and Windows:

240|
    241|
  • WSL accessing Windows files: /mnt/c/, /mnt/d/ mount Windows drives
    • 242|
  • Windows accessing WSL files: Enter \\\\wsl.localhost\\Ubuntu in File Explorer address bar (distro name may differ)
    • 243|
  • Performance advice: Keep project files inside WSL (~/project), don't run them under /mnt/c/ — cross-filesystem IO is slow

5. systemd Service Auto-Start

247|

With systemd enabled, you can write services as .service files that start on boot.

248| 249|

Example: Hermes Dashboard

250| 
sudo tee /etc/systemd/system/hermes-dashboard.service > /dev/null << 'EOF'
251|[Unit]
252|Description=Hermes Agent Dashboard
253|After=network.target
254|
255|[Service]
256|Type=simple
257|User=root
258|ExecStart=/usr/local/bin/hermes dashboard --port 9119 --host 127.0.0.1 --no-open --skip-build
259|Restart=on-failure
260|RestartSec=15
261|
262|[Install]
263|WantedBy=multi-user.target
264|EOF
265|
266|sudo systemctl daemon-reload
267|sudo systemctl enable --now hermes-dashboard
268| 269|
270|

💡 Common Service Management Commands

271| 
# Check status
272|systemctl status service_name --no-pager
273|
274|# View logs
275|journalctl -u service_name -n 50 --no-pager
276|
277|# Restart
278|sudo systemctl restart service_name
279|
280|# Disable auto-start
281|sudo systemctl disable service_name

7. Daily Maintenance Script

310|

Run all maintenance tasks with one command:

311| 
#!/bin/bash
312|# ~/scripts/cleanup.sh
313|set -e
314|
315|echo "=== (1/4) Cleaning apt cache ==="
316|sudo apt autoremove -y
317|sudo apt clean
318|
319|echo "=== (2/4) Cleaning journal logs ==="
320|sudo journalctl --vacuum-time=7d
321|sudo journalctl --vacuum-size=200M
322|
323|echo "=== (3/4) Cleaning pip cache ==="
324|rm -rf ~/.cache/pip/ 2>/dev/null || true
325|uv cache clean 2>/dev/null || true
326|
327|echo "=== (4/4) Disk usage ==="
328|df -h / | tail -1
329|echo "   Done ✅"
330|

Set it to run weekly via cron: crontab -e and add 0 3 * * 0 bash ~/scripts/cleanup.sh &>/dev/null

8. Frequently Asked Questions

333|

For cross-platform development workflows between WSL and .NET, the .NET MAUI development on WSL2 blog post on the Microsoft Dev Blog covers setup and debugging tips.

334| 335|

Q: What is the difference between WSL1 and WSL2?

336|

WSL1 uses a translation layer that maps Linux system calls to Windows, offering fast cross-OS file operations at the cost of compatibility. WSL2 runs a full Linux kernel in a lightweight VM, delivering near-complete Linux compatibility. For most development workflows, especially those involving Docker or systemd, WSL2 is the recommended choice.

337| 338|

Q: WSL startup gives "The referenced object type does not support the attempted operation"

339|

This is usually a virtual NIC driver conflict with proxy software (Clash, V2Ray). Run in admin PowerShell:

340| 
# Fix WSL network driver
341|netsh winsock reset
342|netsh int ip reset all
343|# Reboot your PC to apply
344| 345|

Q: Can ping external network, but curl works

346|

WSL2's ICMP (ping) takes a different path. If your proxy software doesn't support ICMP forwarding, this is normal. Use curl or wget instead of ping for network testing.

347| 348|

Q: WSL disk space keeps growing

349|

The VHDX virtual disk only grows, never shrinks. Follow the steps in section 3 of this guide to compress it periodically.

350| 351|

Q: systemctl shows "Failed to connect to bus"

352|

This means systemd hasn't started. Check if systemd=true is set in /etc/wsl.conf, then run wsl --shutdown and restart WSL.

353| 354|

Q: How do I reset WSL2 completely and reclaim all disk space?

355|

Run wsl --unregister <distro_name> in PowerShell (replace with your distro, e.g., wsl --unregister Ubuntu). This deletes the entire WSL instance, including the VHDX file. Then reinstall with wsl --install -d Ubuntu. Only do this if you have backups of important data.

356| 357|

Q: Can I access WSL2 files from Windows?

358|

Yes. Enter \\\\wsl.localhost\\Ubuntu (or your distro name) in the Windows File Explorer address bar. You can also access Windows files from WSL via /mnt/c/. For performance, always keep project files inside WSL (~/project) rather than on /mnt/c/.
