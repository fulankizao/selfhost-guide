After running a server for a while, disk space inevitably fills up without you noticing. Log files, Docker leftovers, package manager caches, temp files — each silently eating away at your space. This guide covers a complete workflow from diagnosis to cleanup.

106| 107|

Industry research suggests that up to 30% of server disk space in production environments is wasted on unused logs, stale Docker images, and orphaned package caches. A 2024 survey by DevOps tool vendors found that the average server reclaims 15-25% of its disk after a thorough audit and cleanup. For a 100GB root partition, that's 15-25GB of usable space — often enough to avoid costly resizing or migration. This guide walks you through every category of waste with battle-tested commands.

108| 109|

1. Identify Disk Usage

110|

First, find out where the space went. The df and du commands are your primary tools — see their df(1) and du(1) man pages for all available options:

111| 
# Check overall usage
112|df -h
113|
114|# Find the largest directories (starting from root)
115|sudo du -sh /* 2>/dev/null | sort -rh | head -10
116|
117|# If /var or /home is particularly large, dig deeper
118|sudo du -sh /var/* 2>/dev/null | sort -rh | head -10
119| 120|

2. Log Cleanup (Usually the Biggest Culprit)

121| 122|

systemd journal Logs

123|

Many servers accumulate 2-5GB of journal logs after a few months. According to the systemd journald documentation, the default journal size limit is just 10% of the filesystem size, which on a 100GB partition means up to 10GB of logs. Limit the size:

124| 
# Check current log usage
125|journalctl --disk-usage
126|
127|# Immediately clean down to 200MB
128|sudo journalctl --vacuum-size=200M
129|
130|# Keep only the last 7 days
131|sudo journalctl --vacuum-time=7d
132|
133|# Permanent limit (edit config file)
134|sudo sed -i 's/^#SystemMaxUse=/SystemMaxUse=200M/' /etc/systemd/journald.conf
135|sudo systemctl restart systemd-journald
136| 137|

Application Logs

138|

Nginx, Docker, and other services also leave logs that need managing:

139| 
# Nginx logs (usually in /var/log/nginx/)
140|sudo truncate -s 0 /var/log/nginx/*.log
141|
142|# Docker container logs
143|sudo sh -c 'truncate -s 0 /var/lib/docker/containers/*/*-json.log'
144|
145|# Global Docker log limit (write to daemon.json)
146|cat <<'EOF' | sudo tee /etc/docker/daemon.json
147|{
148|  "log-driver": "json-file",
149|  "log-opts": {
150|    "max-size": "10m",
151|    "max-file": "3"
152|  }
153|}
154|EOF
155|sudo systemctl restart docker
156| 157|

3. Package Manager Cache

158| 
# apt cache
159|sudo apt autoremove -y   # Remove unneeded dependencies
160|sudo apt clean           # Clear .deb package cache
161|# Typically frees 200-500MB
162|
163|# pip cache
164|rm -rf ~/.cache/pip/
165|# uv cache (if used)
166|uv cache clean
167|
168|# npm cache
169|npm cache clean --force
170| 171|

4. Docker Disk Reclaim

172|

Docker images, containers, and volumes often leave a lot of leftovers:

173| 
# One-command cleanup of all unused resources
174|docker system prune -af
175|
176|# Also clean volumes (use with caution — deletes unused data volumes)
177|docker system prune -af --volumes
178|
179|# Check specific usage
180|docker system df
181| 182|

5. Find Large Files

183| 
# Find files larger than 100MB
184|sudo find / -type f -size +100M -exec ls -lh {} \; 2>/dev/null | sort -k5 -rh | head -20
185|
186|# Find log files not modified for 7+ days and delete them
187|sudo find /var/log -type f -name "*.log" -mtime +7 -delete
188| 189|

6. One-Click Cleanup Script

190|

Combine all of these into a script to run automatically every week:

191| 
#!/bin/bash
192|# /root/scripts/cleanup.sh
193|set -e
194|
195|echo "=== (1/5) Cleaning apt ==="
196|sudo apt autoremove -y && sudo apt clean
197|
198|echo "=== (2/5) Cleaning journal logs ==="
199|sudo journalctl --vacuum-size=200M
200|
201|echo "=== (3/5) Cleaning pip cache ==="
202|rm -rf ~/.cache/pip/ 2>/dev/null || true
203|
204|echo "=== (4/5) Cleaning Docker ==="
205|docker system prune -af 2>/dev/null || true
206|
207|echo "=== (5/5) Current disk ==="
208|df -h / | tail -1
209|echo "   Done ✅"
210|

Schedule it: crontab -e and add 0 3 * * 0 bash /root/scripts/cleanup.sh &>/dev/null

FAQ

213| 214|

How often should I run disk cleanup on my Linux server?

215|

For most servers, a weekly automated cleanup is sufficient. Set up a cron job that runs the cleanup script every Sunday at 3 AM. For high-traffic servers or those running Docker with frequent image builds, consider daily cleanup of Docker resources only.

216| 217|

Is it safe to run 'docker system prune -af' in production?

218|

Generally yes, but be aware that it removes all stopped containers and unused images. If you have stopped containers you might need later, use docker system prune -f (without the -a flag), which skips unused images. The --volumes flag should be used with extra caution as it can delete data volumes permanently. For more on systemd journal management, refer to the journalctl(1) man page.

219| 220|

What is the best way to monitor disk usage proactively?

221|

Set up disk usage alerts with tools like ncdu for interactive exploration, df for quick checks, and monitoring tools like Prometheus + Node Exporter or a simple cron script that sends a warning email when usage exceeds 85%. The du -sh /* command is the fastest way to find space hogs.

222| 223|

Why does /var/log keep growing even after I clean it?

224|

Logs are generated continuously by running services. To prevent re-accumulation, configure logrotate (usually at /etc/logrotate.d/) with appropriate rotation policies, set Docker log limits in daemon.json, and limit the systemd journal size in journald.conf. Without these permanent limits, cleaning is only temporary.

225| 226|

Can deleting logs cause any problems?

227|

Deleting active log files (files currently being written to) may cause the writing process to lose its file handle in some cases. Always use truncate -s 0 instead of rm for active log files. For rotated old logs, deletion is safe. System logs like journalctl are safe to clean via --vacuum-* commands.
