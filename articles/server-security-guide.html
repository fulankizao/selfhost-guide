  • 10. Summary

Self-hosting gives you full control over your data, services, and infrastructure. But that control comes with a responsibility: you are now the system administrator, and the security of your server is entirely in your hands. A compromised server can leak personal data, serve malware to visitors, or become part of a botnet — all within minutes of going online if basic precautions are skipped.

109|109| 110|110|

According to the Verizon 2025 Data Breach Investigations Report (DBIR), 60% of breaches involved attackers exploiting known vulnerabilities where a patch was available but not applied, and credential-based attacks accounted for over 40% of all breach entry methods. These are the two categories of risk that a well-hardened self-hosted server can virtually eliminate.

111|111| 112|112|

This guide covers the full security stack: SSH hardening, firewall configuration, intrusion prevention with fail2ban, automatic updates, Docker security best practices, AppArmor/SELinux, file integrity monitoring, and log auditing. Each section includes concrete commands you can run on any Ubuntu or Debian-based server.

113|113| 114|114|

1. SSH Hardening — Your First Line of Defense

115|115|

SSH is the most attacked service on any internet-facing Linux server. Automated bots scan the entire IPv4 address space within minutes of a new server coming online, trying default credentials and exploiting old vulnerabilities. Hardening SSH is step zero of server security.

116|116| 117|117|

1.1 Key-Based Authentication Only

118|118|

Generate an ED25519 key pair on your local machine (not the server):

119|119| 
ssh-keygen -t ed25519 -a 100 -f ~/.ssh/selfhost_key
120|120|

Copy the public key to your server:

121|121| 
ssh-copy-id -i ~/.ssh/selfhost_key.pub user@your-server-ip
122|122|

Test that key-based login works, then disable passwords in /etc/ssh/sshd_config:

123|123| 
PasswordAuthentication no
124|124|ChallengeResponseAuthentication no
125|125|PermitRootLogin no
126|126|PubkeyAuthentication yes
127|127|

Apply changes:

128|128| 
sudo systemctl restart sshd
129|129| 130|130|

1.2 Change the Default SSH Port

131|131|

Changing the SSH port from 22 to a high-numbered port (e.g., 2222) drastically reduces log noise from automated scanners:

132|132| 
Port 2222
133|133|

Update /etc/ssh/sshd_config, then add the new port to your firewall (see Section 2). After confirming you can connect on the new port, remove port 22 from your firewall rules.

134|134| 135|135|

1.3 Additional SSH Hardening Options

136|136| 
# In /etc/ssh/sshd_config
137|137|MaxAuthTries 3
138|138|MaxSessions 2
139|139|LoginGraceTime 30
140|140|AllowUsers your-username
141|141|ClientAliveInterval 300
142|142|ClientAliveCountMax 2
143|143|
    144|144|
  • MaxAuthTries 3 — Limits authentication attempts per connection.
    • 145|145|
  • MaxSessions 2 — Prevents a single user from opening too many concurrent sessions.
    • 146|146|
  • LoginGraceTime 30 — Closes unauthenticated connections after 30 seconds.
    • 147|147|
  • AllowUsers — Whitelists specific usernames; everything else is rejected.
    • 148|148|
  • ClientAliveInterval + ClientAliveCountMax — Drops idle connections after 10 minutes of inactivity.
    • 149|149|
150|150| 151|151|

2. Firewall Configuration with UFW

152|152|

The Uncomplicated Firewall (UFW) provides a simple interface to iptables/nftables. A default-deny policy ensures only explicitly allowed ports are reachable.

153|153| 154|154| 
# Set default policies
155|155|sudo ufw default deny incoming
156|156|sudo ufw default allow outgoing
157|157|
158|158|# Allow SSH on your custom port (if changed)
159|159|sudo ufw allow 2222/tcp comment 'SSH'
160|160|
161|161|# Allow HTTP and HTTPS
162|162|sudo ufw allow 80/tcp comment 'HTTP'
163|163|sudo ufw allow 443/tcp comment 'HTTPS'
164|164|
165|165|# Allow your self-hosted services (e.g., WireGuard)
166|166|sudo ufw allow 51820/udp comment 'WireGuard'
167|167|
168|168|# Enable the firewall
169|169|sudo ufw enable
170|170|
171|171|# Check the rules
172|172|sudo ufw status verbose
173|173|

UFW logs blocked packets to /var/log/ufw.log. Check it periodically to spot scanning activity:

174|174| 
sudo tail -f /var/log/ufw.log | grep BLOCK
175|175|

Pro tip: Never enable UFW over SSH before allowing your SSH port — you will lock yourself out. Always add your SSH port rule first, then enable.

176|176| 177|177|

2.1 Rate Limiting with UFW

178|178|

UFW supports basic rate limiting to mitigate brute-force attempts:

179|179| 
sudo ufw limit ssh
180|180|

This allows 6 connections per 30 seconds from the same IP before blocking further attempts for that period.

3. Fail2ban — Automated Intrusion Prevention

183|183|

Fail2ban monitors service logs for repeated failure patterns and temporarily bans offending IPs via firewall rules. It ships with pre-configured filters for SSH, Apache, Nginx, and many other services.

184|184| 185|185| 
# Install fail2ban
186|186|sudo apt update
187|187|sudo apt install fail2ban -y
188|188|
189|189|# Create a local configuration file
190|190|sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local
191|191|

Edit /etc/fail2ban/jail.local to customize settings:

192|192| 
[DEFAULT]
193|193|bantime = 3600          # 1 hour ban (in seconds)
194|194|findtime = 600          # 10 minute window
195|195|maxretry = 5            # 5 failures triggers a ban
196|196|
197|197|[sshd]
198|198|enabled = true
199|199|port = 2222             # Match your custom SSH port
200|200|

Add a jail for Nginx (if you use it):

201|201| 
[nginx-http-auth]
202|202|enabled = true
203|203|
204|204|[nginx-botsearch]
205|205|enabled = true
206|206|

Start fail2ban:

207|207| 
sudo systemctl enable fail2ban --now
208|208|
209|209|# Check banned IPs
210|210|sudo fail2ban-client status sshd
211|211|

Fail2ban is especially valuable for self-hosters because it protects services that must be publicly accessible (like web apps) from credential stuffing and directory traversal attacks.

4. Automatic Security Updates

214|214|

The Verizon 2025 DBIR finding that 60% of breaches involved unpatched vulnerabilities underscores the critical importance of keeping your system up to date. Automate this with unattended-upgrades:

215|215| 216|216| 
sudo apt install unattended-upgrades -y
217|217|sudo dpkg-reconfigure --priority=low unattended-upgrades
218|218|

For fine-grained control, edit /etc/apt/apt.conf.d/50unattended-upgrades:

219|219| 
Unattended-Upgrade::Allowed-Origins {
220|220|    "${distro_id}:${distro_codename}-security";
221|221|};
222|222|
223|223|Unattended-Upgrade::AutoFixInterruptedDpkg "true";
224|224|Unattended-Upgrade::MinimalSteps "true";
225|225|Unattended-Upgrade::Remove-Unused-Dependencies "true";
226|226|Unattended-Upgrade::AutomaticReboot "false";
227|227|

For Docker containers, use Watchtower to automatically update running containers:

228|228| 
docker run -d \
229|229|  --name watchtower \
230|230|  --restart unless-stopped \
231|231|  -v /var/run/docker.sock:/var/run/docker.sock \
232|232|  containrrr/watchtower \
233|233|  --schedule "0 0 4 * * *"   # 4 AM daily
234|234|

Warning: Review update-testing on a staging instance before enabling automatic container restarts for production services.

5. Docker Security Best Practices

237|237|

Docker adds a layer of isolation, but misconfiguration can actually weaken security. Follow these rules for every container you run:

238|238| 239|239|

5.1 Run as Non-Root User

240|240|

Many official Docker images default to root. Override this in your docker-compose.yml:

241|241| 
services:
242|242|  myapp:
243|243|    image: myapp:latest
244|244|    user: "1000:1000"   # UID:GID of a non-root user
245|245| 246|246|

5.2 Drop All Capabilities, Add Only What's Needed

247|247|

Linux capabilities grant privileged operations to containers. Drop all and add only what your app requires:

248|248| 
services:
249|249|  nginx:
250|250|    image: nginx:alpine
251|251|    cap_drop:
252|252|      - ALL
253|253|    cap_add:
254|254|      - NET_BIND_SERVICE  # Allows binding to port 80
255|255| 256|256|

5.3 Read-Only Root Filesystem

257|257|

Prevent attackers from writing malicious files to the container's filesystem:

258|258| 
services:
259|259|  myapp:
260|260|    image: myapp:latest
261|261|    read_only: true
262|262|    tmpfs:
263|263|      - /tmp   # Allow temp writes only in tmpfs
264|264| 265|265|

5.4 Never Mount the Docker Socket

266|266|

Mounting /var/run/docker.sock into a container gives it root-level access to the Docker daemon — equivalent to full host compromise. Avoid this pattern unless you fully understand the implications. If you must (e.g., for Watchtower or Portainer), restrict it to a dedicated management container with minimal privileges.

267|267| 268|268|

5.5 Use Docker Bench Security

269|269|

Run Docker Bench Security to audit your host and container configuration against the CIS Docker Benchmark:

270|270| 
docker run --rm --net host \
271|271|  -v /etc:/etc:ro \
272|272|  -v /var/run/docker.sock:/var/run/docker.sock:ro \
273|273|  docker/docker-bench-security
274|274|

This script checks 100+ security controls and highlights misconfigurations with remediation steps.

6. Mandatory Access Control: AppArmor & SELinux

277|277|

Linux Discretionary Access Control (DAC) — standard file permissions — is the first security layer. Mandatory Access Control (MAC) systems like AppArmor (Ubuntu default) or SELinux (Fedora/CentOS default) add a second layer that constrains what processes can do even when running as root.

278|278| 279|279|

AppArmor profiles are pre-installed for common services. Check their status:

280|280| 
sudo aa-status
281|281|

Enable AppArmor for a specific Docker container by appending a security-opt flag:

282|282| 
services:
283|283|  myapp:
284|284|    image: myapp:latest
285|285|    security_opt:
286|286|      - apparmor:docker-default
287|287|

On SELinux-based systems, ensure it is enforcing:

288|288| 
getenforce
289|289|# Should output: Enforcing
290|290|
291|291|# Set to enforcing if it's permissive
292|292|sudo setenforce 1
293|293|

MAC systems would have prevented many high-profile container escape vulnerabilities — the extra layer catches exploits that bypass standard permissions.

294|294| 295|295|

7. File Integrity Monitoring with AIDE

296|296|

If an attacker does gain access, you need to know what changed. AIDE (Advanced Intrusion Detection Environment) builds a database of file hashes and alerts you when files are modified.

297|297| 298|298| 
# Install AIDE
299|299|sudo apt install aide -y
300|300|
301|301|# Initialize the database (first run only)
302|302|sudo aideinit
303|303|
304|304|# Move the database to its default location
305|305|sudo mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db
306|306|
307|307|# Run a check
308|308|sudo aide --check
309|309|

Schedule weekly checks via cron:

310|310| 
sudo crontab -e
311|311|# Add this line:
312|312|0 3 * * 0 /usr/bin/aide --check | mail -s "AIDE report" your@email.com
313|313| 314|314|

8. Log Auditing and Centralized Monitoring

315|315|

Logs tell the story of what happened. Without a monitoring system, you are flying blind.

316|316| 317|317|

8.1 Auditd for System Call Monitoring

318|318|

Auditd records system calls, file accesses, and authentication events:

319|319| 
sudo apt install auditd -y
320|320|sudo auditctl -e 1   # Enable auditing
321|321|sudo aureport --summary   # View summary report
322|322|

Monitor /etc/passwd for unauthorized changes:

323|323| 
sudo auditctl -w /etc/passwd -p wa -k passwd_changes
324|324|sudo auditctl -w /etc/shadow -p wa -k shadow_changes
325|325|

View audit logs with ausearch:

326|326| 
sudo ausearch -k passwd_changes --start today
327|327| 328|328|

8.2 Centralized Logging with Rsyslog

329|329|

Forward logs to a remote log server (or a syslog-compatible service) so attackers cannot tamper with them after gaining access:

330|330| 
# In /etc/rsyslog.conf
331|331|*.* @logs.example.com:514   # UDP forwarding
332|332|# Or over TCP with TLS for sensitive environments
333|333| 334|334|

8.3 Integration with Monitoring Tools

335|335|

Pair security hardening with a monitoring stack from our Monitoring Your Homelab guide. Tools like Grafana, Prometheus, and Loki can visualize failed SSH attempts, firewall drops, and system resource anomalies that may indicate compromise.

9. Additional Hardening Measures

338|338| 339|339|

9.1 Disable Unused Services

340|340|

Every running service is a potential attack surface. List listening ports and audit them:

341|341| 
sudo ss -tulpn   # List all listening TCP and UDP ports
342|342|sudo lsof -i -P -n | grep LISTEN
343|343|

Disable and mask any service you don't need:

344|344| 
sudo systemctl disable --now avahi-daemon
345|345|sudo systemctl mask avahi-daemon
346|346| 347|347|

9.2 Kernel Hardening with sysctl

348|348|

Add these lines to /etc/sysctl.d/99-hardening.conf:

349|349| 
# IP spoofing protection
350|350|net.ipv4.conf.all.rp_filter = 1
351|351|net.ipv4.conf.default.rp_filter = 1
352|352|
353|353|# Ignore ICMP redirects
354|354|net.ipv4.conf.all.accept_redirects = 0
355|355|net.ipv6.conf.all.accept_redirects = 0
356|356|
357|357|# Ignore source-routed packets
358|358|net.ipv4.conf.all.accept_source_route = 0
359|359|net.ipv6.conf.all.accept_source_route = 0
360|360|
361|361|# Disable IPv6 if not needed, or only enable privacy extensions
362|362|net.ipv6.conf.all.use_tempaddr = 2
363|363|
364|364|# Increase SYN backlog and enable SYN cookies (DoS protection)
365|365|net.ipv4.tcp_syncookies = 1
366|366|net.ipv4.tcp_syn_backlog = 1024
367|367|

Apply immediately:

368|368| 
sudo sysctl -p /etc/sysctl.d/99-hardening.conf
369|369| 370|370|

9.3 Two-Factor Authentication for SSH

371|371|

For an additional layer beyond SSH keys, enable TOTP-based 2FA using libpam-google-authenticator:

372|372| 
sudo apt install libpam-google-authenticator -y
373|373|google-authenticator    # Follow the setup, scan QR code into your authenticator app
374|374|

Edit /etc/pam.d/sshd to add:

375|375| 
auth required pam_google_authenticator.so
376|376|

Then in /etc/ssh/sshd_config:

377|377| 
AuthenticationMethods publickey,keyboard-interactive
378|378|

Now SSH requires both your private key and a time-based one-time password — defense in depth at its finest.

10. Summary

381|381|

Securing a self-hosted server is not a one-time task — it is an ongoing practice. The measures in this guide form a layered defense that protects against automated attacks, credential theft, unpatched vulnerabilities, and container escapes.

382|382|

Here is a deployment checklist for a new self-hosted server:

383|383|
    384|384|
  • Disable SSH password authentication; use ED25519 keys only
    • 385|385|
  • Change the SSH port and restrict by AllowUsers
    • 386|386|
  • Set up UFW with a default-deny policy
    • 387|387|
  • Install and configure fail2ban for SSH and web services
    • 388|388|
  • Enable unattended-upgrades for automatic security patches
    • 389|389|
  • Audit Docker containers: non-root user, drop capabilities, read-only fs
    • 390|390|
  • Enable AppArmor or verify SELinux is enforcing
    • 391|391|
  • Initialize AIDE for file integrity monitoring
    • 392|392|
  • Configure auditd and forward logs to a remote destination
    • 393|393|
  • Run Docker Bench Security and lynis for a final audit
    • 394|394|
395|395|

Implement these steps in order, testing each change before moving to the next. Automate where possible — cron jobs for AIDE checks, Watchtower for Docker updates, and periodic lynis audits ensure your server stays hardened even when you are not actively managing it.

396|396|

As the Verizon 2025 DBIR reminds us, the overwhelming majority of breaches exploit known vulnerabilities and weak credentials. By eliminating both categories, you move your self-hosted server from an easy target to a hardened system that automated attackers will skip in favor of easier prey.

FAQ

399|399| 400|400|

What is the most important security step for a new self-hosted server?

401|401|

Disabling password-based SSH authentication and using SSH key pairs only. This single measure blocks the vast majority of automated brute-force attacks. Combined with changing the default SSH port and installing fail2ban, you eliminate 99% of opportunistic attacks before they reach your services.

402|402| 403|403|

Do I need a firewall if my server is behind a home router?

404|404|

Yes. A host-based firewall like UFW or nftables provides defense-in-depth. Even behind a NAT router, internal threats (compromised IoT devices on your LAN, malware, or a misconfigured service binding to 0.0.0.0) can expose ports to your local network. A firewall ensures only the services you intend to expose are reachable.

405|405| 406|406|

How often should I update my self-hosted server?

407|407|

Security patches should be applied within 24-48 hours of release for critical vulnerabilities. Unattended-upgrades can handle this automatically for apt packages. Docker containers should be rebuilt and redeployed weekly via a cron job or watchtower. The Verizon 2025 Data Breach Investigations Report found that 60% of breaches involved unpatched known vulnerabilities where a patch was available but not applied.

408|408| 409|409|

Is fail2ban still necessary if I use SSH keys?

410|410|

Yes. While SSH keys eliminate password-guessing attacks on SSH, fail2ban protects other services too — web apps, mail servers, and any service that accepts authenticated connections. It monitors log files for repeated failures and temporarily bans offending IPs via the firewall.

411|411| 412|412|

Should I use Docker for better security isolation?

413|413|

Docker provides significant isolation benefits when configured correctly. Each container runs in its own namespaces with limited capabilities by default. However, Docker is not a security panacea — you must avoid privilege escalation with --privileged flags, use read-only root filesystems where possible, drop unnecessary Linux capabilities, and never mount the Docker socket into containers unless absolutely required.
