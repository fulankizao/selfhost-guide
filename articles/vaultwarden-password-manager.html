Setting Up a Self-Hosted Password Manager with Vaultwarden

2026-06-16 · 12 min read · views
← Back to Home

The average internet user has over 100 online accounts, according to a 2023 NordPass study on digital habits. Yet most people reuse the same 3-4 passwords across all of them. The 2024 Verizon Data Breach Investigations Report found that roughly 31% of all breaches involve credential theft or misuse — a figure that has remained stubbornly high year after year because weak and reused passwords are the easiest attack vector for adversaries.

A password manager solves this problem at its root. It generates unique, complex passwords for every account, stores them in an encrypted vault, and autofills them when you need to log in. But trusting a third-party cloud service with all your passwords requires a leap of faith. What if their server is breached? What if they go out of business? What if their privacy policy changes? Self-hosting your own password manager gives you full control over your credential data without sacrificing convenience.

This guide walks you through deploying Vaultwarden — a lightweight, Rust-based server implementation that is fully API-compatible with all official Bitwarden clients. You will set up a production-ready password manager on your own server using Docker, secure it with a reverse proxy and TLS, and configure SMTP so you can invite family members or teammates.

1. Why Vaultwarden Instead of Bitwarden?

Bitwarden is an excellent open-source password manager. Its cloud service is secure, audited, and reasonably priced at $10/year for premium features. However, Bitwarden's official self-hosted offering — the self-hosted Bitwarden server — is a relatively heavy deployment. It requires Docker with multiple containers (MSSQL database, web server, API, identity server, and admin console), uses up to 2 GB of RAM on startup, and has disk requirements of 30 GB or more. Many of the premium features require an active paid license even when self-hosted.

Vaultwarden (formerly known as bitwarden_rs) solves these problems. Written in Rust, it compiles to a single binary that consumes as little as 10-20 MB of RAM at idle and needs only a few hundred MB of disk storage. It implements the entire Bitwarden API suite — vault sync, TOTP codes, attachments, send, passkeys/FIDO2, organizations, collections, emergency access — and all premium features are unlocked for free. No license keys, no hidden paywalls.

Vaultwarden is not an official Bitwarden project, but it has earned wide trust in the self-hosting community. The software is actively maintained, receives regular security updates, and is deployed on tens of thousands of servers worldwide. Its GitHub repository has over 40,000 stars and a responsive community that quickly addresses issues.

2. Prerequisites

Before you start, you will need:

  • A Linux server (Ubuntu 22.04 or Debian 12 recommended) with at least 1 GB RAM and 10 GB disk
  • Docker and Docker Compose installed (see our Docker Compose guide)
  • A domain name (e.g., vault.yourdomain.com) with DNS pointing to your server
  • A reverse proxy — Traefik, Nginx Proxy Manager, or Caddy — handling TLS termination
  • Ports 80 and 443 open in your firewall
  • An SMTP service (SendGrid, Mailgun, or your own mail server) for invitation and password reset emails

You do not need a dedicated VPS. Vaultwarden runs comfortably on a Raspberry Pi 4 or a low-tier cloud VM (e.g., a $5/month Linode or DigitalOcean droplet). Its minimal resource footprint means you can even run it alongside other services on the same machine.

3. Deploying Vaultwarden with Docker Compose

The cleanest way to run Vaultwarden is with Docker Compose. Create a directory for the project and add the following docker-compose.yml:

mkdir -p ~/vaultwarden
cd ~/vaultwarden
nano docker-compose.yml
version: '3'

services:
  vaultwarden:
    image: vaultwarden/server:latest
    container_name: vaultwarden
    restart: unless-stopped
    volumes:
      - ./vw-data:/data
    environment:
      - DOMAIN=https://vault.yourdomain.com
      - SIGNUPS_ALLOWED=false
      - INVITATIONS_ALLOWED=true
      - ADMIN_TOKEN=your-strong-admin-token-here
      - SMTP_HOST=smtp.your-email-provider.com
      - SMTP_FROM=vaultwarden@yourdomain.com
      - SMTP_PORT=587
      - SMTP_SECURITY=starttls
      - SMTP_USERNAME=your-smtp-username
      - SMTP_PASSWORD=your-smtp-password
      - WEBSOCKET_ENABLED=true
    ports:
      - "127.0.0.1:8080:80"
    networks:
      - proxy

networks:
  proxy:
    external: true

Key configuration points:

  • DOMAIN — Must match the exact public URL of your instance. This is critical for WebSocket connections (live sync) and attachment downloads.
  • SIGNUPS_ALLOWED=false — Disable open registration. Users can only join via email invitation, which is far more secure.
  • INVITATIONS_ALLOWED=true — Lets you invite team members or family.
  • ADMIN_TOKEN — A strong random string (use openssl rand -base64 48 to generate one). This protects the admin panel at /admin.
  • WEBSOCKET_ENABLED=true — Enables real-time vault sync across devices. Without this, clients must manually refresh or wait for periodic syncs.
  • ports: "127.0.0.1:8080:80" — Binds to localhost only. The reverse proxy (running on the same Docker network) will route public traffic to it.

The networks: proxy: external: true line assumes you already have a reverse proxy container running on a shared Docker network named proxy. If you do not have one yet, you can either create the network first (docker network create proxy) or skip the network binding and expose Vaultwarden directly with Let's Encrypt via the Vaultwarden built-in ACME support.

4. Setting Up the Reverse Proxy

Vaultwarden must be served over HTTPS. Here is a quick Traefik configuration example (add a dynamic config file or router rule in your Traefik setup):

# Traefik dynamic config (YAML)
http:
  routers:
    vaultwarden:
      rule: "Host(`vault.yourdomain.com`)"
      entryPoints:
        - websecure
      service: vaultwarden
      tls:
        certResolver: letsencrypt

  services:
    vaultwarden:
      loadBalancer:
        servers:
          - url: "http://vaultwarden:80"

If you use Nginx Proxy Manager or Caddy, the setup is similar. The key requirement is that your reverse proxy forwards requests to the vaultwarden container on port 80 and that the Host header (vault.yourdomain.com) matches the DOMAIN environment variable exactly.

After configuring the reverse proxy, start Vaultwarden:

docker compose up -d
docker compose logs -f  # watch for startup messages

If everything is working, you will see log output confirming that the server is listening on port 80 and that the admin panel is accessible at https://vault.yourdomain.com/admin (authenticated by your ADMIN_TOKEN).

5. Initial Setup and User Management

With the server running, follow these steps to get your first user onboarded:

  1. Enable signups temporarily — Set SIGNUPS_ALLOWED=true in your docker-compose.yml and run docker compose up -d to restart. Create your own admin account through the web vault at https://vault.yourdomain.com.
  2. Disable signups and invite others — After creating your account, set SIGNUPS_ALLOWED=false back and restart. Use the admin panel at /admin to invite users by email. Each invited user receives an email with a registration link and can set their own master password.
  3. Configure organizations (optional) — From the web vault, create an Organization for family or team shared vaults. Invite other users and assign them to collections with read/write or read-only permissions.
  4. Enable two-factor authentication — Every user should set up 2FA on their account. Vaultwarden supports authenticator app (TOTP), Duo, WebAuthn (FIDO2 security keys), and YubiKey. The master password is the single point of failure — 2FA adds a critical second layer.

Important security note: Your master password is the key to the entire vault. Bitwarden clients use a zero-knowledge architecture — the server never receives your master password in plaintext. Instead, the client derives an encryption key using PBKDF2 SHA-256 (or Argon2id) with 600,000 KDF iterations (configurable up to 2,000,000 for Argon2). This means even if an attacker gains full access to your Vaultwarden database, they would need to brute-force your master password through these iterations to decrypt your vault. Choose a master password that is long (at least 16 characters), high-entropy, and never used anywhere else.

6. Backing Up Your Vault

All Vaultwarden data — encrypted vaults, attachments, user settings — is stored in the ./vw-data directory you mounted in the Docker Compose file. This directory contains an SQLite database (db.sqlite3), attachment files, and icon cache.

Your backup strategy should include:

  • Regular SQLite dumps — Run sqlite3 vw-data/db.sqlite3 ".backup /path/to/backup/vaultwarden-$(date +%F).db" as a cron job. This produces a consistent snapshot without downtime.
  • Full directory backup — Copy the entire vw-data directory to off-site storage (rsync to another server, rclone to Backblaze B2 or S3, or simply to encrypted cloud storage).
  • Test restoration — Periodically restore a backup to a test directory and verify that users can log in and their vault contents are intact. A backup you never test is a backup you cannot trust.
  • Export a plaintext JSON backup — From the admin panel or web vault settings, export your vault. Store this encrypted (GPG) off-site as a last-resort fallback.

Refer to our Backup Strategies for Self-Hosted Services guide for a comprehensive discussion of backup rotation, encryption, and off-site storage methods.

7. Connecting Clients

One of Vaultwarden's greatest strengths is that it works with every official Bitwarden client without modification. Here is how to connect each one:

Browser Extensions

Install the Bitwarden extension for Chrome, Firefox, Edge, or Safari. Open the extension settings, scroll to Self-hosted environment, enter your server URL (https://vault.yourdomain.com), and save. Log in with your email and master password — the extension will sync your vault and start autofilling immediately.

Desktop Apps

Download Bitwarden for Windows, macOS, or Linux. At the login screen, click the gear icon, switch from Bitwarden Cloud to Self-hosted, and enter your server URL. The desktop app supports offline access — your vault is cached locally and remains accessible even when the server is unreachable.

Mobile Apps (Android / iOS)

Install the Bitwarden mobile app from Google Play or the App Store. Tap the gear icon on the login screen, select Self-hosted, and enter your server URL. Mobile apps can enable biometric unlock (fingerprint or Face ID) so you do not need to type your master password every time. Vaultwarden also supports FIDO2 WebAuthn for passwordless login on supported mobile browsers.

CLI

The bitwarden-cli Node.js tool is useful for scripting and automation:

npm install -g @bitwarden/cli
bw config server https://vault.yourdomain.com
bw login
bw unlock  # exports session key
bw list items  # list all vault items

9. Security Best Practices

Self-hosting a password manager means you are responsible for its security. Follow these practices to keep your vault safe:

  • Use a strong ADMIN_TOKEN — generate it with openssl rand -base64 48 and store it in a password manager (ironically, a different one until your Vaultwarden is running). The admin panel gives access to user management and vault exports — treat the token like a root password.
  • Keep Vaultwarden updated — pull the latest Docker image regularly: docker compose pull && docker compose up -d. The Vaultwarden project publishes security patches promptly. Subscribe to the GitHub releases feed or set up Watchtower to auto-update.
  • Enforce strong master passwords — Vaultwarden allows configuring minimum password complexity and KDF iterations. Set PASSWORD_ITERATIONS=600000 in your environment to enforce high PBKDF2 iterations for all users.
  • Use rate limiting — Vaultwarden has built-in rate limiting to prevent brute-force attacks on login and 2FA endpoints. Configure it with LOGIN_RATELIMIT_MAX_BURST and LOGIN_RATELIMIT_SECONDS if needed.
  • Monitor logs — Check docker compose logs periodically for failed login attempts, admin panel access, and suspicious activity. Feed logs into a centralized logging system like Loki or Papertrail if you manage multiple services.
  • Disable admin panel remotely — If you rarely use the admin panel, set up a firewall rule or reverse proxy ACL to allow /admin access only from your home IP range or VPN subnet.

10. Advanced Configuration

Vaultwarden supports many additional features through environment variables. Some useful ones:

  • DISABLE_ICON_DOWNLOAD=true — Disables fetching website favicons for vault entries. This saves bandwidth and avoids DNS lookups for every saved URL. Icons are a minor convenience; disabling them has no security impact.
  • ORG_CREATION_USERS=all — Who can create organizations. Set to none if only the admin should create shared vaults.
  • EMERGENCY_ACCESS_ALLOWED=true — Enables the Emergency Access feature, allowing trusted users to request access to your vault after a configurable waiting period. Useful for family members who need access to financial accounts if something happens to you.
  • SENDS_ALLOWED=true — Enables Bitwarden Send, a secure file/text sharing feature similar to Firefox Send. Files are encrypted before upload and self-destruct after being viewed.

For the complete list of configuration options, see the Vaultwarden Wiki on GitHub.

FAQ

Can I migrate from another password manager to Vaultwarden?

Yes. Vaultwarden supports importing data from over 40 different password managers and formats, including Bitwarden (JSON/CSV), LastPass, 1Password, Dashlane, KeePass, and Chrome/Firefox/Safari built-in password managers. Go to the web vault, navigate to Tools > Import Data, select your source format, and upload the exported file. Most imports preserve folders, notes, and custom fields.

What happens if my self-hosted server goes offline?

Your vault data is cached locally on every client. Browser extensions, desktop apps, and mobile apps all store an encrypted copy of your vault locally. You can still unlock, view, copy, and autofill passwords when offline. Changes you make while offline are queued and synced to the server when connectivity is restored. The only thing you lose during an outage is the ability to add new items from a device that has no local cache.

Can I run Vaultwarden behind Cloudflare Tunnel?

Yes, this is a popular setup because it eliminates the need to open ports on your firewall. Configure Cloudflare Tunnel (cloudflared) on the same machine as Vaultwarden, point the tunnel public hostname to localhost:8080, and Cloudflare handles TLS termination automatically. Note that WebSocket support (required for live sync) must be enabled in the Cloudflare dashboard — it is enabled by default for free plans. See our Cloudflare Tunnel guide for detailed instructions.

Does Vaultwarden support hardware security keys (YubiKey, Nitrokey)?

Yes. Vaultwarden supports FIDO2/WebAuthn for both 2FA and passwordless authentication. You can register hardware security keys as a second factor alongside TOTP or as a standalone passwordless login method. This works with YubiKey, Nitrokey, SoloKey, and most FIDO2-compliant security keys. Configuration is done per-user in the web vault under Settings > Security > Two-Step Login.

How often should I update Vaultwarden?

Check for updates at least once a month. The Vaultwarden project releases new versions roughly every 2-4 weeks, with security patches published immediately. You can use docker compose pull && docker compose up -d to update, or set up Watchtower to do it automatically. Always review the release notes before updating, as breaking changes are occasionally introduced.

Summary

Self-hosting a password manager with Vaultwarden is one of the highest-impact improvements you can make to your digital security posture. It eliminates your reliance on third-party cloud services for credential storage, gives you complete control over your encryption keys and data, and provides a seamless experience across every device you own.

Here is the quick-start checklist:

  • Set up a Linux server with Docker and Docker Compose
  • Create a docker-compose.yml with Vaultwarden, configured with a strong ADMIN_TOKEN and SMTP credentials
  • Configure a reverse proxy (Traefik, Nginx, or Caddy) with a valid TLS certificate from Let's Encrypt
  • Start the container, create your admin account, then disable open signups
  • Invite family or team members, set up organizations and collections
  • Install Bitwarden clients on all devices and point them to your self-hosted server URL
  • Enable 2FA on every account, especially the admin user
  • Set up automated backups of the vw-data directory
  • Check for updates monthly and keep the Docker image current

The 2024 Verizon DBIR data is clear: credential theft remains the top attack vector. A password manager with unique, high-entropy passwords for every site is your best defense. Self-hosting that manager with Vaultwarden means you do not have to trust anyone else with the keys to your digital kingdom.
